## Section 1: Memory Requirements
**Does your user return across multiple sessions?**
- [x] Yes — the same user will return and expect the agent to remember them.

**If yes, how long should the agent remember a user?**
- [x] Until the tab closes (session storage).

**What is your estimate of average conversation length?**
- [x] 10–30 turns.

## Section 2: Technical Decisions
**Which session storage approach are you implementing today?**
- [x] In-memory (process variable) — using the `_sessions` dictionary in `session_service.py`.

**What is your trim strategy?**
- [x] Sliding window — keep system prompt + last N messages.

**What is your window size (N messages)?**
We will keep the last 40 messages (20 turns).
Rationale: This fits comfortably within the Gemini Flash context window while preserving enough history for complex asset refinement.

**Where does the session ID come from?**
- [x] Generated by the frontend using `uuid.uuid4()` and stored in Streamlit session state.

## Section 3: Known Limitations
- [x] Sessions are lost when the server restarts or redeploys.
- [x] Sessions cannot be shared across multiple server instances.

**What is your plan to address these before Demo Day?**
We will migrate to a Redis backend in Week 12 to ensure session persistence across server restarts and scale.
